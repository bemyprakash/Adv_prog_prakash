1. Project Overview & Objectives
The Food Delivery System is a comprehensive, object-oriented software application designed to simulate a real-world platform similar to Zomato, Swiggy, or UberEats. The system manages the entire lifecycle of a food order — from browsing menus and placing an order, to live delivery tracking and post-order customer support.

1.1 Project Objectives
•	Design a scalable, modular system using OOP principles (Inheritance, Encapsulation, Abstraction, Polymorphism).
•	Implement role-based access for five distinct actors: Customer, Restaurant Owner, Delivery Agent, Customer Support, and Admin.
•	Include a complete Customer Support ticketing module to handle real-world exceptions such as failed payments, wrong orders, and profile issues.
•	Provide full design documentation including UML class diagrams, use case diagrams, flow diagrams, and sequence diagrams.
•	Produce production-grade code architecture suitable for extension into a fully functional application.

2. System Architecture & Actors
2.1 System Actors
The system defines five distinct actor roles. Each actor interacts with the system through a well-defined interface, and four of the five actors share a common base class, demonstrating the inheritance design pattern.

Actor	Role Description	Key Responsibilities
Customer	End-user who orders food	Browse menu, place/cancel orders, track delivery, rate, raise complaints
Restaurant Owner	Business owner who receives orders	Update menu, accept/reject orders, mark preparation status
Delivery Agent	Field personnel who delivers orders	Accept delivery, update GPS location, send emergency alerts
Customer Support	Internal support representative	Handle complaints, resolve payments, escalate tickets, update profiles
Admin	System administrator	Manage all users, view reports, configure system settings

2.2 High-Level Architecture
The system follows a layered architecture: Presentation Layer (UI) → Business Logic Layer (Classes & OOP) → Data Layer (Database / Storage). The core of the business logic layer is the inheritance hierarchy rooted in the UserAccount base class.

PRESENTATION LAYER:  Customer App  |  Restaurant Portal  |  Agent App  |  Support Dashboard  |  Admin Panel
         |                    |                     |                  |                      |             
BUSINESS LOGIC:  UserAccount (Base)  →  Customer / DeliveryAgent / RestaurantOwner / CustomerSupport
                 Order  →  Payment  →  SupportTicket  →  Rating  →  Notification             
DATA LAYER:      MySQL / PostgreSQL  —  ORM / Repository Pattern                             

3. OOP Concepts Applied
OOP Pillar	How It Is Applied In This Project	Example
Inheritance	Customer, DeliveryAgent, RestaurantOwner, CustomerSupport all extend UserAccount, inheriting userId, name, phone, email, login() and logout().	class Customer extends UserAccount
Encapsulation	All class attributes are private/protected. Access is via getter/setter methods. SupportTicket hides its internal state (Open → In Progress → Resolved) from external classes.	ticket.updateStatus("Resolved")
Abstraction	The Customer does not know HOW a complaint is processed internally. They only call raiseComplaint(); the backend CustomerSupport module handles logic invisibly.	customer.raiseComplaint("Payment failed")
Polymorphism	All user types extend UserAccount. A reference of type UserAccount can point to any derived object. Each subclass overrides or adds its own behaviour (e.g., different dashboard actions).	UserAccount u = new Customer()

5.2 Detailed Use Case: Place Order (UC03)
Use Case ID	UC03
Use Case Name	Place Order
Primary Actor	Customer
Description	Customer selects items from a restaurant menu, applies an optional coupon code, and confirms an order that is sent to the restaurant.
Precondition	Customer is logged in. Restaurant menu is loaded. Customer has at least one item in cart.
Postcondition	Order object is created with status PENDING. Payment is initiated. Restaurant is notified.
Main Flow	1. Customer browses menu and adds items to cart.  2. Customer optionally enters coupon code; system validates and applies discount.  3. Customer selects delivery address and payment method.  4. Customer taps 'Place Order'; system creates Order object.  5. Payment is processed (see UC05).  6. Restaurant receives notification. Order status changes to CONFIRMED.

5.3 Detailed Use Case: Raise Support Ticket (UC08)
Use Case ID	UC08
Use Case Name	Raise Support Ticket
Primary Actor	Customer
Description	Customer experiences an issue (e.g., payment failure, wrong item, late delivery) and creates a support ticket through the app.
Precondition	Customer is logged in. Issue relates to an existing order or payment transaction.
Postcondition	SupportTicket object is created with status OPEN. A CustomerSupport agent is notified and assigned.
Main Flow	1. Customer navigates to 'Help & Support' section.  2. Customer selects issue type: Payment / Order / Profile / Other.  3. Customer enters description and links relevant orderId.  4. System creates SupportTicket with unique ticketId and status OPEN.  5. System notifies available CustomerSupport agent.  6. Agent is assigned; ticket status changes to IN_PROGRESS.  7. Agent resolves issue; status changes to RESOLVED. Customer is notified.
 
6. System Flow Diagrams
6.1 Order Placement Flow
1	Customer logs in via app and browses restaurant listings.
2	Customer selects a restaurant → views menu items → adds items to cart.
3	Customer applies coupon code (optional). System validates code and applies discount.
4	Customer reviews cart, selects delivery address and payment method.
5	Customer confirms order. System creates Order object with status = PENDING.
6	Payment is processed. On success → status = CONFIRMED. On failure → SupportTicket raised automatically.
7	Restaurant is notified. Owner accepts order → status = ACCEPTED → PREPARING.
8	System assigns nearest available DeliveryAgent to the order.
9	Agent picks up order. Status = OUT_FOR_DELIVERY. Customer can track live location.
10	Agent delivers order. Status = DELIVERED. Customer prompted to rate.

6.2 Customer Support Flow (New Module)
1	Customer experiences an issue (payment failed / wrong order / item missing / profile error).
2	Customer opens Help & Support in app. Selects issue type from dropdown.
3	Customer fills description form and links orderId or transactionId.
4	System creates SupportTicket object: ticketId generated, status = OPEN, timestamp recorded.
5	System auto-assigns ticket to available CustomerSupport agent (by availabilityStatus).
6	Agent opens ticket. Reviews order/payment data. Status updated to IN_PROGRESS.
7	Agent takes action: issues refund via Payment.refund(), corrects profile, or contacts restaurant.
8	If issue is complex, agent calls escalateIssue() → Admin is notified.
9	Issue resolved. Agent sets status = RESOLVED. Customer receives in-app notification.
10	Customer confirmation requested. If unsatisfied, ticket can be reopened.

6.3 Delivery Agent Emergency Flow
1	DeliveryAgent is on route. Experiences emergency (accident, road block, vehicle breakdown).
2	Agent opens app and taps Emergency Update button.
3	Agent calls sendEmergencyUpdate(message, location).
4	System notifies Admin and CustomerSupport immediately.
5	System sends automated message to Customer explaining delay.
6	Admin re-assigns a new DeliveryAgent to the pending order.
7	New agent accepts delivery. Customer is notified of reassignment.

9.3 Java Class Skeleton (CustomerSupport)
public class CustomerSupport extends UserAccount {
    private String supportId;
    private List<SupportTicket> assignedTickets;
    private String availabilityStatus;  // 'AVAILABLE', 'BUSY', 'OFFLINE'

    public void handleComplaint(SupportTicket ticket) {
        ticket.updateStatus(TicketStatus.IN_PROGRESS);
        // Review linked Order and Payment
    }

    public void resolvePaymentIssue(String ticketId) {
        SupportTicket t = getTicket(ticketId);
        Payment p = PaymentService.getByOrderId(t.getOrderId());
        if (p.getStatus() == PaymentStatus.FAILED) {
            p.refund(p.getAmount());
            t.updateStatus(TicketStatus.RESOLVED);
        }
    }

    public void escalateIssue(String ticketId) {
        AdminService.notifyAdmin(ticketId);  // escalate to Admin
    }
}

10.3 Conclusion
The Food Delivery System project demonstrates a thorough application of advanced programming principles within a realistic, commercially-relevant domain. By using a proper inheritance hierarchy rooted at UserAccount, all four user-facing roles (Customer, DeliveryAgent, RestaurantOwner, CustomerSupport) share a consistent base, enabling polymorphic behaviour and clean code reuse.

The addition of the CustomerSupport module elevates the system beyond a basic delivery app into a fully operational platform. The SupportTicket class introduces a ticket-based workflow modelled after real-world systems such as Zomato Support and Freshdesk, demonstrating scalable and modular issue resolution with clearly defined state transitions (OPEN → IN_PROGRESS → RESOLVED).

The system is designed for extensibility: future modules such as real-time chat between Customer and Support, AI-based ticket triage, a Refund class, and a Notification service can all be added without restructuring the existing architecture.

KEY OOP DESIGN DECISIONS SUMMARY

  Inheritance:     UserAccount → Customer, DeliveryAgent, RestaurantOwner, CustomerSupport
  Encapsulation:   SupportTicket hides status transitions; Payment hides transaction logic
  Abstraction:     Customer calls raiseComplaint() without knowing backend ticket processing
  Polymorphism:    Any UserAccount reference can hold Customer, Agent, or Support object

  'The ticket-based support system mirrors industry-standard platforms, ensuring scalability,
   modularity, and professional-grade exception handling in a real-world OOP application.'
